OpenGL Project — Structure Plan

Keeping it tidy as shaders, objects, and experiments grow

Current Pain Points

What gets messy fast

  • Raw VAO/VBO/EBO handles loose in main()
  • All shaders flat in one folder
  • Uniform setting scattered everywhere in the render loop
  • No separation between "setup" and "per-frame" logic
  • Adding a new object = copy-pasting 20 lines of buffer setup

What to do instead

  • Wrap GPU resources in a Mesh class
  • One shader subfolder per experiment/scene
  • Render loop just binds + draws, doesn't manage state
  • Later: hot-reload shaders from file without recompiling C++

Step 1 — Mesh Class do now

Wraps VAO, VBO, EBO. Construction uploads data to GPU. Destruction cleans up. You never touch raw handles in main again.

struct Mesh {
    unsigned int VAO, VBO, EBO;
    int indexCount;

    // stride: number of floats per vertex (e.g. 8 for pos+normal+uv)
    Mesh(float* verts, int vertBytes,
         unsigned int* indices, int idxCount,
         int stride)
    {
        this->indexCount = idxCount;
        glGenVertexArrays(1, &VAO);
        glGenBuffers(1, &VBO);
        glGenBuffers(1, &EBO);

        glBindVertexArray(VAO);
        glBindBuffer(GL_ARRAY_BUFFER, VBO);
        glBufferData(GL_ARRAY_BUFFER, vertBytes, verts, GL_STATIC_DRAW);
        glBindBuffer(GL_ELEMENT_ARRAY_BUFFER, EBO);
        glBufferData(GL_ELEMENT_ARRAY_BUFFER,
                     idxCount * sizeof(unsigned int), indices, GL_STATIC_DRAW);

        glVertexAttribPointer(0, 3, GL_FLOAT, GL_FALSE,
                              stride * sizeof(float), (void*)0);
        glEnableVertexAttribArray(0);
    }

    void draw() const {
        glBindVertexArray(VAO);
        glDrawElements(GL_TRIANGLES, indexCount, GL_UNSIGNED_INT, 0);
    }

    ~Mesh() {
        glDeleteVertexArrays(1, &VAO);
        glDeleteBuffers(1, &VBO);
        glDeleteBuffers(1, &EBO);
    }

    // non-copyable — GPU resources shouldn't be duplicated by accident
    Mesh(const Mesh&) = delete;
    Mesh& operator=(const Mesh&) = delete;
};

Your render loop then looks like:

riemannShader.use();
riemannShader.setMat4("model", modelSphere);
riemannShader.setVec2("mouse", mx, my);
sphere.draw();

flatShader.use();
flatShader.setMat4("model", modelPlane);
plane.draw();

Step 2 — Shared GLSL Library do now

Your complex math functions (cMul, cDiv, cExp…) are going to be useful in every fragment shader you write. Instead of copy-pasting them, pull them into shaders/shared/complex.glsl and #include them at load time — your Shader class can do a simple string substitution when it sees #include "shared/complex.glsl". 

// In Shader constructor, before glShaderSource:
// replace lines starting with #include "X" with the contents of file X
std::string source = readFile(path);
source = resolveIncludes(source, shaderDir); // ~20 lines

Step 3 — Shader Hot-Reload next

Instead of recompiling C++ every time you change a function, watch the .glsl files for changes and recompile the shader program on the fly. This also gives you a poor-man's "function input" — just edit the file, save, and the sphere updates in under a second. 

// In render loop:
if (fileChanged("shaders/riemann/fragment.glsl")) {
    riemannShader.reload(); // recompile + relink
}

fileChanged just compares std::filesystem::last_write_time — one line per file per frame, negligible cost.

Later (when you actually feel the pain) later

Scene list

When you have 5+ objects, a std::vector<SceneObject> where each holds a Mesh + Shader pointer + model matrix, and the render loop just iterates it, is the next natural step.

Framebuffers / render targets

When you want post-processing (bloom on the sphere, render-to-texture) you'll need FBOs. Add them only then.

UBO / uniform blocks

When you have many shaders all needing the same camera matrices, a Uniform Buffer Object lets you set them once and all shaders see them. Not worth it until you have 3+ shaders.

The rule: only add abstraction when you feel the pain of not having it.
