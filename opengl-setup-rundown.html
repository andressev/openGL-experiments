OpenGL Window Setup

A full rundown of every step taken to get an OpenGL window running in a C++ CMake project on Windows.

1 The Three Things You Need

OpenGL on Windows requires three separate pieces before you can draw anything:

Window & Input
GLFW

Creates the OS window and an OpenGL rendering context. Without it you'd need to write hundreds of lines of raw Win32 code. Also handles keyboard, mouse, and gamepad input.

Function Loader
GLAD

Windows only ships with OpenGL 1.1 headers from 1997. GLAD generates C code that asks the GPU driver at runtime for pointers to every modern OpenGL function (like glClear, glDrawArrays, etc.).

Already on your machine
opengl32.lib

Ships with every Windows SDK and MSVC install. You just tell CMake to link against it — no download needed.

2 Why C API ≠ C Only

OpenGL's API is written in C, but C++ is fully backwards-compatible with C. You call C functions directly from .cpp files with zero friction. Most real-world OpenGL code is written in C++.

3 Project Structure After Setup

CMakeProjectTest/
  CMakeLists.txt  ← root build config
  CMakePresets.json  ← x64/x86 debug/release presets
  CMakeProjectTest/
    CMakeProjectTest.cpp  ← your code
    CMakeProjectTest.h
    CMakeLists.txt  ← sub-project build config (edited)
  external/  ← third-party libs live here
    glfw/
      include/GLFW/  ← glfw3.h
      lib-vc2022/  ← glfw3.lib (static, no DLL needed)
    Glad/
      include/glad/  ← glad.h
      include/KHR/  ← khrplatform.h
      src/  ← glad.c (compiled as part of your exe)

The external/ folder is just a convention — a regular folder that keeps third-party code separate from your own. You could call it vendor/ or libs/, it makes no difference to the compiler.

4 CMakeLists.txt Changes

Only the sub-project file (CMakeProjectTest/CMakeLists.txt) needed editing:

add_executable (CMakeProjectTest
    "CMakeProjectTest.cpp"
    "CMakeProjectTest.h"
    "../external/Glad/src/glad.c"   # glad.c compiles as plain C alongside your C++ files
)

set_property(TARGET CMakeProjectTest PROPERTY CXX_STANDARD 20)

target_include_directories(CMakeProjectTest PRIVATE
    "../external/glfw/include"        # so #include <GLFW/glfw3.h> works
    "../external/Glad/include"        # so #include <glad/glad.h> works
)

target_link_libraries(CMakeProjectTest PRIVATE
    "${CMAKE_SOURCE_DIR}/external/glfw/lib-vc2022/glfw3.lib"
    opengl32                          # Windows built-in, no path needed
)

What each part does

glad.c in add_executable

GLAD is a C file, not a library. It just gets compiled directly into your executable.

target_include_directories

Tells the compiler where to search when it sees #include <...>. Without this, headers won't be found.

glfw3.lib

Static library — the GLFW code is baked into your .exe. No .dll file needs to be shipped alongside it.

opengl32

The low-level Windows OpenGL interface. CMake finds it automatically, you just name it.

5 The Source Code, Line by Line

// glad MUST come before glfw — it defines the OpenGL symbols glfw depends on
#include <glad/glad.h>
#include <GLFW/glfw3.h>
#include <iostream>

int main()
{
    glfwInit();                                          // start GLFW
    glfwWindowHint(GLFW_CONTEXT_VERSION_MAJOR, 3);      // request OpenGL 3.3
    glfwWindowHint(GLFW_CONTEXT_VERSION_MINOR, 3);
    glfwWindowHint(GLFW_OPENGL_PROFILE, GLFW_OPENGL_CORE_PROFILE); // Core = no legacy stuff

    GLFWwindow* window = glfwCreateWindow(800, 600, "OpenGL Window", nullptr, nullptr);
    if (window == nullptr) {
        std::cout << "Failed to create GLFW window" << std::endl;
        glfwTerminate();
        return -1;
    }
    glfwMakeContextCurrent(window);   // make this window the active OpenGL context

    if (!gladLoadGLLoader((GLADloadproc)glfwGetProcAddress)) {
        std::cout << "Failed to initialize GLAD" << std::endl; // load all OpenGL functions
        return -1;
    }

    // render loop — runs every frame until the window is closed
    while (!glfwWindowShouldClose(window)) {
        glClearColor(0.2f, 0.3f, 0.3f, 1.0f);  // set background color (RGBA)
        glClear(GL_COLOR_BUFFER_BIT);             // wipe the screen

        glfwSwapBuffers(window);   // show the frame (double buffering)
        glfwPollEvents();           // process keyboard/mouse/close events
    }

    glfwTerminate();   // clean up all GLFW resources
    return 0;
}

Key concepts

glfwWindowHint

Sets options before the window is created. Here we're requesting OpenGL 3.3 Core — modern features, no deprecated legacy API.

gladLoadGLLoader

Queries the GPU driver for every OpenGL function pointer. Must happen after glfwMakeContextCurrent, because it needs an active context to talk to the driver.

glfwSwapBuffers

Double buffering: you draw to a hidden back buffer, then swap it to the screen at once. This prevents flickering.

glfwPollEvents

Checks for OS events (close button, key presses). Without this the window freezes and becomes unresponsive.

6 The CMake Reconfigure Trick

After deleting the out/ folder (the build cache), CMake needs to reconfigure from scratch before it can build. In Visual Studio the fix is:

Project → Delete Cache and Reconfigure — or just close and reopen the folder. Visual Studio automatically reruns CMake on open.

This is safe to do any time you change a CMakeLists.txt file or add new dependencies.

7 What's Next

Shaders

Everything drawn on screen goes through vertex and fragment shaders — small programs that run on the GPU. This is the core of modern OpenGL.

VBO / VAO

Vertex Buffer Objects and Vertex Array Objects are how you send geometry (triangles, etc.) to the GPU.

GLM

A header-only math library (vectors, matrices, transforms) that mirrors GLSL syntax. Drop it in external/ like you did with GLFW and GLAD.
