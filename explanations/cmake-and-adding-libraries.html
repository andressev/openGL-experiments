CMakeLists.txt

How it works, what every command means, and how to add new libraries — including GLM.

1 The Big Picture

CMake is not a compiler. It is a recipe translator. You describe what you want to build in CMakeLists.txt, and CMake translates that into build files that a real build tool (Ninja) understands. Ninja then calls MSVC (cl.exe) to actually compile.

CMakeLists.txt  ──►  CMake  ──►  build.ninja  ──►  Ninja + cl.exe  ──►  .exe
  (you write)                    (generated,                          (final output)
                                  lives in out/)

You never touch the generated files. You only ever edit CMakeLists.txt.

2 Two Files, Two Jobs

Your project has two CMakeLists.txt files. Each folder that participates in the build has one.

Root
CMakeProjectTest/CMakeLists.txt

Global setup only — project name, CMake version requirement, and which subfolders to include. Think of it as the table of contents.

Sub-project
CMakeProjectTest/CMakeProjectTest/CMakeLists.txt

The real build instructions — what files to compile, which headers to find, which libraries to link. This is where you spend most of your time.

3 Root File — Command by Command

CMakeLists.txt (root)
cmake_minimum_required (VERSION 3.10)
# If the machine running cmake is older than 3.10, stop with an error.
# Prevents confusing failures on outdated setups.

project ("CMakeProjectTest")
# Names the project. Also sets variables like CMAKE_PROJECT_NAME.
# You reference this name in IDEs, logs, and install scripts.

add_subdirectory ("CMakeProjectTest")
# Tells CMake to go read CMakeLists.txt inside that subfolder.
# Add one line per subfolder that has its own build logic.

4 Sub-project File — Command by Command

CMakeProjectTest/CMakeLists.txt
add_executable (CMakeProjectTest
    "CMakeProjectTest.cpp"
    "CMakeProjectTest.h"
    "../external/Glad/src/glad.c"
)
# Creates a TARGET named CMakeProjectTest.
# A target is the thing you're building — an .exe in this case.
# Every .cpp and .c listed here gets compiled and linked together.
# .h files are optional here (VS uses them for IntelliSense only).

set_property(TARGET CMakeProjectTest PROPERTY CXX_STANDARD 20)
# Sets the C++ language standard for this target.
# 20 = C++20. Could also be 17, 14, 11.

target_include_directories(CMakeProjectTest PRIVATE
    "../external/glfw/include"
    "../external/Glad/include"
)
# Adds folders to the compiler's header search path.
# When your code says #include <GLFW/glfw3.h>, the compiler
# looks in these folders to find it.
# PRIVATE = only this target uses these paths.

target_link_libraries(CMakeProjectTest PRIVATE
    "${CMAKE_SOURCE_DIR}/external/glfw/lib-vc2022/glfw3.lib"
    opengl32
)
# Hands .lib files to the linker so it can resolve function calls
# that were declared in headers but live in pre-compiled libraries.
# CMAKE_SOURCE_DIR = the root folder of your project (auto-set by CMake).

5 Understanding PRIVATE / PUBLIC / INTERFACE

You'll see these keywords on every target_include_directories and target_link_libraries call. They control who inherits the setting.

PRIVATE
Only this target

The most common choice. Use it when the library is an internal implementation detail and nothing else needs to know about it.

PUBLIC
This target + anything that links to it

Use when you're building a library that exposes the dependency in its own headers. Consumers automatically inherit it.

INTERFACE
Only consumers, not this target

Used for header-only libraries. The library itself doesn't compile anything, but whoever includes it needs the headers.

For a standalone executable like yours, PRIVATE is always correct. Executables don't get linked into anything else, so PUBLIC and INTERFACE have no effect.

6 The Three Types of Libraries You'll Encounter

Header-only
Just headers, no .lib file

The entire library is in .h files. You only need target_include_directories. GLM is this type — drop the folder in, point CMake at it, done.

Static library
.lib — baked into your .exe

Pre-compiled code that gets copied directly into your executable at link time. No extra files needed to run the program. GLFW (glfw3.lib) is this type.

Source library
.c / .cpp — compiled with your code

Just raw source files you add directly to add_executable. GLAD is this type — glad.c compiles right alongside your own files.

7 Adding GLM (The Math Library)

GLM is header-only — the easiest type to add. No .lib, no .c file. Just headers.

1
Download GLM

Get it from GitHub (search "g-truc glm releases") and extract it into external/glm/. The folder should contain a glm/ subfolder full of .hpp files.

2
Add one line to CMakeLists.txt

Just add its include path. No lib to link, no source file to compile.

target_include_directories(CMakeProjectTest PRIVATE
    "../external/glfw/include"
    "../external/Glad/include"
    "../external/glm"             # ← add this line
)
3
Use it in your code

That's all CMake needs. Now include GLM headers in your .cpp files.

#include <glm/glm.hpp>
#include <glm/gtc/matrix_transform.hpp>

glm::vec3 position(0.0f, 0.0f, 3.0f);   // 3D vector
glm::mat4 model = glm::translate(glm::mat4(1.0f), position);  // 4x4 matrix

8 General Recipe for Adding Any Library

Follow this decision tree every time you add a new library:

1
Put it in external/

Create a subfolder like external/libraryname/ and extract the library there.

2
Add the include path

Always needed. Add the folder containing the headers to target_include_directories.

3
Header-only? You're done.

Libraries like GLM need nothing more after step 2.

4
Has a .lib file? Link it.

Add the path to the .lib file in target_link_libraries.

5
Has source files (.c/.cpp)? Add them to add_executable.

Like GLAD's glad.c — list the source file directly alongside your own files.

9 Useful CMake Variables

${CMAKE_SOURCE_DIR}

The root folder of your project — where the top-level CMakeLists.txt lives. Use this for absolute paths to files.

${CMAKE_CURRENT_SOURCE_DIR}

The folder of the CMakeLists.txt currently being processed. Useful in subdirectory files.

${CMAKE_BUILD_TYPE}

Either Debug or Release depending on the preset selected. You can use it in if() blocks.

${CMAKE_VERSION}

The version of CMake running. Used in the if(CMAKE_VERSION VERSION_GREATER ...) guard in your file.

10 After Every CMakeLists.txt Change

CMake needs to re-run whenever you edit a CMakeLists.txt. In Visual Studio this happens automatically when you save the file — you'll see "CMake generation started" in the output window. If it doesn't trigger, use:

Project → Configure CMakeProjectTest — forces a reconfigure manually.

You do not need to delete out/ for normal changes. Only delete it if CMake gets into a broken state and reconfiguring doesn't fix it.
